先把你这个站现在“是什么”说清楚：**heartopiagame.com 目前本质上是一个“下载落地页”**——用户点完下载就离开，因为站内没有“下载前/下载后立刻用得上”的内容闭环。你要加的模块必须满足你给的硬约束：**(1) 一个炫酷的数据展示（数据一次性好拿）+ (2) 一个轻量高频小功能（一次性拉全量数据到本地 JSON），并且两者都不能用 AI。**

下面给你两个模块设计（每个都有宣传语），并把数据来源、获取方式、API/JSON 规格写清楚。

---

## 模块 1（数据展示｜炫酷｜一次性好拿）：「官方更新雷达 Official Update Radar」

### 它展示什么（为什么用户爱看）

**展示“官方最近说了什么/改了什么/活动到哪了”**——这类信息在新游/上线期是最常被问的：开服、补偿、活动、联动、维护、规则变更。你的 Steam 商店页有 Heartopia 的 AppID（4025700）。([Steam 商店][1])

> 宣传语（首页标题 + 副标题）

* **标题**：官方更新雷达
* **副标题**：**30 秒扫完最近更新：维护、活动、福利、重要改动，一屏看懂。**

### 炫酷怎么做（视觉方案，不靠 AI）

做一个“雷达面板”（Dashboard）——全靠简单统计 + 关键词字典（不是大模型）：

* **时间轴 Timeline**：按日期把公告卡片排开（支持筛选：维护/活动/福利/联动）。
* **热度脉冲 Pulse**：最近 30 天公告数量折线/柱状（越密集说明越“忙”）。
* **话题标签 Chips**：用你写死的关键词字典把公告打标签（例：`maintenance`、`event`、`reward`、`collab`、`gacha`、`bugfix`），并统计出现次数。
* **一条“今日重点”**：默认置顶最新一条公告标题 + 3 行摘要（摘要用规则截断 maxlength，不用 AI）。

### 数据怎么“轻松一次性获取”（官方 API）

用【Steamworks Web API】的【ISteamNews/GetNewsForApp v2】拉官方新闻/公告。官方文档给了接口与参数说明。([partner.steamgames.com][2])

**请求 URL（你在构建时跑一次，生成本地 JSON）**

* Endpoint：`https://api.steampowered.com/ISteamNews/GetNewsForApp/v2/` ([partner.steamgames.com][2])
* 示例（把 appid 固定为 4025700）：
  `?appid=4025700&count=50&maxlength=300&format=json`
* 可选参数（常用）：

  * `count`：取多少条（默认 20）([partner.steamgames.com][2])
  * `maxlength`：内容最大长度（0 表示全量；也可以给 300 做摘要）([partner.steamgames.com][2])
  * `enddate`：拉某个时间点之前的（做翻页/历史）([partner.steamgames.com][2])
  * `feeds`：只取某类 feed（很多人用 `steam_community_announcements`）([Stack Overflow][3])

> 场景示范（怎么用这串工具）

1. 打开【ISteamNews 文档页】看参数含义（appid/count/maxlength）。([partner.steamgames.com][2])
2. 在你项目构建脚本里请求一次接口，把返回写到：`/public/data/steam_news.json`
3. 首页直接读取这个本地 JSON 渲染“雷达面板”（页面不会频繁打外网请求）

### 输出 JSON 格式（你本地落盘建议格式）

Steam 返回的核心字段大致是这些（gid/title/url/contents/date/feedlabel 等）。([Go Packages][4])

你落到本地可以规范成这样（便于前端炫酷展示）：

```json
{
  "source": "steam_api",
  "appid": 4025700,
  "fetchedAt": "2026-01-13T00:00:00Z",
  "items": [
    {
      "gid": "512979846355223182",
      "title": "Heartopia Community Update & Steam Release Notice",
      "url": "https://store.steampowered.com/news/app/4025700/view/...",
      "date": 1736,
      "feedlabel": "Community Announcements",
      "contentsPreview": "..."
    }
  ]
}
```

## 模块 2（轻量小功能｜高频刚需｜不靠 AI）：「兑换码奖励筛选器 Code Finder」

### 它解决什么痛点（为什么能把人留住）

**兑换码是典型“高频、低成本、强动机”需求**：玩家不想翻 3 个网页比对、复制、还怕过期。你把它做成“筛选 + 一键复制 + 标注到期”，用户下载后会顺手点两下。

> 宣传语

* **标题**：兑换码奖励筛选器
* **副标题**：**只看你想要的奖励：月光水晶 / 许愿星 / 修理包……一键复制，顺手领完再走。**

### 用户输入 / 输出 / 处理逻辑（纯规则，不用 AI）

**输入（尽量点击/筛选，降低成本）**

* 奖励类型筛选：`Moonlight Crystals` / `Wishing Stars` / `Repair Kits` / `Gold` / `Materials`
* “只看未过期”开关
* “紧急到期”开关（比如 7 天内到期）
* 搜索框（可选）：输入 code 片段或奖励关键词

**输出**

* 符合条件的 codes 列表：`code + 奖励 + 到期日 + 来源 + 复制按钮`
* 置顶提醒：**“最先领这些（最快过期）”**

**处理逻辑**

* 页面加载本地 `codes.json`
* 过滤：按 rewardType / expiresAt / keyword
* 排序：默认按到期日升序（越急越靠前）
* “可信度标识”：同一 code 若同时出现在 2 个来源，就打 ✅（只是集合求交，不是 AI）

### 数据怎么获取（一次性拉全量到本地 JSON）

你可以用“系统性攻略站/媒体攻略页”做来源，不抓论坛。

强烈建议把【heartopia.gg/codes】当主数据源：它把**分类、总数、到期日**写得很结构化，并标注了最近验证时间（2026-01-12）。([Heartopia.gg][5])
同时用【PCGamesN】或【Pocket Tactics】做交叉验证（两者也在 2026-01-12 更新）。([PCGamesN][6])

> 场景示范（怎么用这串工具）

1. 你的构建脚本（或你本地手动跑一次）抓取【heartopia.gg/codes】页面，把“code/奖励/到期日”解析出来。([Heartopia.gg][5])
2. 可选：再抓一次【PCGamesN】或【Pocket Tactics】做对照，给相同 code 打 ✅。([PCGamesN][6])
3. 生成并保存：`/public/data/heartopia_codes.json`
4. 首页“Code Finder”只读本地 JSON，不会频繁请求外部站点

### 你本地 codes.json 建议格式

（支持筛选、到期排序、来源对照）

```json
{
  "fetchedAt": "2026-01-13T00:00:00Z",
  "primarySource": "https://heartopia.gg/codes",
  "codes": [
    {
      "code": "dcthx4u",
      "rewards": [{"item": "Wishing Star", "qty": 10}],
      "expiresAt": "2026-06-30",
      "category": "Discord 300K",
      "sources": ["heartopia.gg", "pcgamesn"],
      "sourceUrls": ["https://heartopia.gg/codes", "https://www.pcgamesn.com/heartopia/codes"]
    }
  ]
}
```

* 下载链接用新标签打开（用户不会被带走）
* 下载完成提示区放 3 个下一步按钮：**【领兑换码】/【看官方更新】/【新手 10 分钟上手】**

### 你这两块为什么符合你的硬约束

* **数据展示**：官方公告/更新，用户常讨论；来源是【Steamworks Web API】；一次性拉取即可。([partner.steamgames.com][2])
* **轻量小功能**：兑换码筛选，高频刚需；来源是结构化攻略页（非论坛），一次性抓取落本地 JSON。([Heartopia.gg][5])
* 两个功能都只做**规则解析/过滤/排序**，不碰 AI。

如果你愿意把“炫酷程度”再拉满一档：我会建议“官方更新雷达”用 **时间轴 + 关键词热力条 + ‘近 30 天游玩重点’三栏布局**（仍然只靠字典和计数），视觉上会非常像“游戏作战指挥台”，很能留人。
